API란 무엇인가

API(Application Programming Interface)는 소프트웨어 애플리케이션 간의 상호 작용을 가능하게 하는 매개체이다. API는 특정 기능을 수행하거나 데이터를 교환할 수 있도록 정의된 규칙과 방법을 제공한다. 이를 통해 개발자는 복잡한 기능을 직접 구현하지 않고도 다양한 서비스를 통합하고 확장할 수 있다. 예를 들어, 지우가 만들었던 혼잡도 어플리케이션. 혼잡도를 구하는 로직을 직접 만들필요 없이 혼잡도 기능을 제공하는 API를 통합하여 어플리케이션을 구현할 수 있다.

REST API란

REST(Representational State Transfer) API는 HTTP 프로토콜을 기반으로 하는 API이다. REST는 웹 표준에 기반한 구조적 스타일로, 클라이언트와 서버 간의 상호 작용을 단순화하고 효율적으로 만든다. REST API는 자원을 표현하고 조작하는 데 있어 일관된 인터페이스를 제공하여, 다양한 클라이언트가 서버와 원활하게 통신할 수 있게 한다.

REST API 규칙

REST API는 다음과 같은 기본 규칙을 따른다:

  1. 자원(Resource) 기반: 모든 데이터는 고유한 URI(Uniform Resource Identifier)를 통해 접근한다. 자원은 서버에서 관리되는 데이터나 서비스를 의미하며, 각각의 자원은 고유한 URI를 갖는다. 예를 들어, https://api.example.com/users는 사용자 자원을 나타내며, 특정 사용자 자원(유저 번호 123)은 https://api.example.com/users/123와 같은 URI를 통해 접근할 수 있다. 다른 예로는 https://api.example.com/products는 상품 자원을, https://api.example.com/orders는 주문 자원을 나타낸다.
  2. HTTP 메서드: 자원에 대한 작업은 HTTP 메서드를 통해 정의된다.
    • GET: 자원의 조회
    • POST: 자원의 생성
    • PUT: 자원의 전체 수정
    • PATCH: 자원의 부분 수정
    • DELETE: 자원의 삭제

위의 자원 기반 규칙과 HTTP 메서드를 조합하여 아래와 같은 예시를과 같이 API요청이 표현될 수 있다.
상품 추가는 POST: /products,
특정 게시판(3번 게시판)에 게시글 추가는 POST: /boards/3/articles
특정 댓글 게시글(17번 게시글) 삭제는 DELETE: /articles/17

  1. 무상태성(Stateless): 각 요청은 독립적이며, 서버는 요청 간의 상태를 저장하지 않는다. 클라이언트는 필요한 모든 정보를 요청에 포함해야 한다. (다음에 설명해줄게)
  2. 표준화된 응답 구조: 응답은 일관된 포맷(JSON, XML 등)으로 제공되어야 하며, 상태 코드를 통해 요청의 결과를 명확히 전달해야 한다.
    • 200 OK: 성공
    • 201 Created: 자원 생성 성공
    • 400 Bad Request: 잘못된 요청
    • 401 Unauthorized: 인증 필요 (로그인이 안된 경우)
    • 401 Forbidden: 권한 없음 (로그인은 되어있으나, 특정 작업을 수행하기 위한 ‘권한’이 없는 경우)
    • 404 Not Found: 자원 없음
    • 500 Internal Server Error: 서버 오류

훨씬 많지만 크게 중요한건 요거, 요거는 외우면 좋아요.

Controller, Service, Repository

소프트웨어 아키텍처에서 Controller, Service, Repository 패턴은 코드의 역할을 분리하여 관리성과 확장성을 높이는 데 사용한다.

API 서버의 전체적인 흐름은 위의 그림과 같다. 이 구조는 특정 언어나 프레임워크의 종속적이지 않으며, 일반적인 구조이다.

Controller

Controller는 사용자 입력을 받아서 처리하고, 적절한 서비스 메서드를 호출한다. 주로 HTTP 요청을 받아서 응답을 반환하는 역할을 한다. 예를 들어, 웹 애플리케이션에서 사용자 로그인 요청을 처리하는 Controller는 사용자가 입력한 자격 증명을 받아서 서비스 계층에 전달하고, 그 결과를 바탕으로 응답을 반환한다.

입력에 대한 처리는 예를들어, 아까 설명했던 것처럼 게시글 작성할때 작성자를 현재 로그인된 유저로 설정해준다던지 등등이 있을 수 있다. 상황에 따라 다양한데 하다보면 무슨 말인지 알게 될거야

Service

Service 계층은 비즈니스 로직을 처리하는 곳이다. 여러 Repository를 조합하여 데이터를 처리하고, 필요한 비즈니스 규칙을 적용한다. 예를 들어, 주문 처리 서비스는 상품 재고를 확인하고, 결제를 처리하며, 주문을 데이터베이스에 저장하는 등 다양한 비즈니스 로직을 수행한다.

이게 일반적으로 비지니스 로직이라는 terminology를 사용하는데, 나도 이게 왜그런지는 모르겠고 비지니스 로직은 그냥 사용자의 요구사항을 처리하는 로직 정도로 생각하면 될 것 같아.

DB의 데이터가 필요하거나 엑세스(CRUD)가 필요한 경우, Repository에게 요청한다. 프로젝트의 규모가 커질 수록 서비스 레이어의 내용이 가장 비대해 질 것이다.

Repository

Repository는 데이터 액세스를 담당하는 계층이다. 데이터베이스와의 상호 작용을 관리하며, 데이터를 조회, 저장, 삭제하는 메서드를 제공한다. 예를 들어, 사용자 정보를 저장하고 조회하는 Repository는 데이터베이스에서 사용자 데이터를 관리하는 모든 작업을 수행한다.

실제로 SQL쿼리를 작성하고 이 SQL쿼리를 실행시켜 데이터베이스의 연산을 수행한다.

ORM (Object-Relational Mapping)

ORM은 객체 지향 프로그래밍 언어를 사용하여 관계형 데이터베이스를 조작할 수 있게 하는 기술이다. ORM은 데이터베이스 테이블을 클래스와 객체로 매핑하여, SQL 쿼리를 작성하지 않고도 데이터베이스 작업을 수행할 수 있게 한다.
그래서 우리는 Sequelize라는 ORM을 사용해서 직접 SQL을 작성하지 않고 객체지향 코드를 작성하듯이 간단하게 데이터베이스에 접근할 수 있다.

Repository 계층을 ORM으로 대체하기

우리처럼 일부 프로젝트에서는 Repository 계층을 ORM으로 대체할 수 있다. 예를 들어, Sequelize와 같은 ORM을 사용하면 데이터베이스 작업을 쉽게 처리할 수 있다. ORM을 사용하면 데이터베이스와의 직접적인 상호 작용을 줄이고, 코드의 일관성을 유지할 수 있다.

장점

  • 빠른 개발: 기본적인 CRUD 작업을 빠르게 구현할 수 있다.
  • 모듈화: 데이터베이스 모델을 모듈화하여 재사용할 수 있다.
  • 생산성 향상: 반복적인 SQL 쿼리 작성 작업을 줄여 개발 속도를 높인다.
  • 유지 보수 용이: 코드의 가독성과 유지 보수성을 높인다. 데이터베이스 스키마와 코드 간의 일관성을 유지할 수 있다.
  • 데이터베이스 독립성: 특정 데이터베이스에 종속되지 않고, 다양한 DBMS를 사용할 수 있게 한다.

단점

  • 복잡성: 단순한 쿼리에 비해 성능이 떨어질 수 있으며, 복잡한 쿼리 작성이 어려울 수 있다.
  • 추상화 비용: ORM이 제공하는 추상화 레이어로 인해 직접 SQL을 사용할 때보다 성능이 저하될 수 있다.
  • 제한적인 쿼리: 복잡한 쿼리 작성 시 ORM의 기능적 한계가 있을 수 있다.
